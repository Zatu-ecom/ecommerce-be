# Product & File Module Integration Pre-Spec

## 1. Overview
This document outlines the architectural decisions and integration strategy for linking the newly developed **File Module** with the **Product Module**. A typical e-commerce product has multiple images (e.g., a primary hero image, gallery thumbnails) and potentially documents (e.g., user manuals, warranty PDFs). 

The goal is to define how the Product module stores references to these files, and exactly how the data should be structured when returning Product payloads to the frontend.

---

## 2. Data Model Integration

To maintain strict domain boundaries in our modular monolith, the Product module will **not** store hard database foreign keys pointing to the File module's `file_object` primary keys. Instead, it will store the UUIDv7 `file_id`.

### Proposed Product Database Tables

```sql
-- Product Image Association Table
CREATE TABLE product_image (
    id BIGSERIAL PRIMARY KEY,
    product_id BIGINT NOT NULL REFERENCES product(id) ON DELETE CASCADE,
    file_id VARCHAR(36) NOT NULL, -- UUIDv7 referencing File Module
    is_primary BOOLEAN DEFAULT FALSE,
    display_order INT DEFAULT 0,
    created_at TIMESTAMPTZ DEFAULT NOW(),
    
    -- Ensure only one primary image per product
    UNIQUE (product_id, file_id)
);
```

## 3. API Response Design Options

When the client requests `GET /api/products/:id` or `GET /api/products`, we must attach the images. Since we recently added `includeDownloadUrl=true` to the File Module's `GetAllFiles` internal method, we have three primary ways to return this data to the frontend:

### Option A: Just the URLs (Minimalist)
The product payload strips all file metadata and just returns an array of direct public URLs.
```json
{
  "id": 101,
  "name": "Wireless Headphones",
  "images": [
    "https://s3.amazonaws.com/bucket/hero.jpg",
    "https://s3.amazonaws.com/bucket/gallery1.jpg"
  ]
}
```
* **Pros:** Extremely lightweight, very easy for the frontend to render `<img src="..." />`.
* **Cons:** The frontend has no `fileId` to use if they want to delete, reorder, or update an image later. They also lose access to the responsive `variants` (thumbnails).

### Option B: Full FileItem Metadata (Maximalist)
We return the exact `FileItem` struct generated by the File Module, embedded inside the Product response.
```json
{
  "id": 101,
  "name": "Wireless Headphones",
  "images": [
    {
      "fileId": "018e6b...",
      "purpose": "PRODUCT_IMAGE",
      "status": "ACTIVE",
      "originalFilename": "hero.jpg",
      "downloadUrl": "https://s3.amazonaws.com/bucket/hero.jpg",
      "variants": [
         { "variantCode": "thumb_200", "downloadUrl": "..." }
      ],
      "isPrimary": true,       // Injected by Product module
      "displayOrder": 0        // Injected by Product module
    }
  ]
}
```
* **Pros:** Frontend gets everything. They can build responsive `srcset` image tags using the `variants`, and they have the `fileId` for CRUD operations.
* **Cons:** The payload is larger, containing fields the storefront might not care about (like `storageProvider`, `purpose`, `status`).

### Option C: Hybrid Product Image DTO (Recommended)
We map the internal `FileItem` into a domain-specific `ProductImageDTO` that abstracts away the underlying storage details but keeps necessary identifiers and URLs.
```json
{
  "id": 101,
  "name": "Wireless Headphones",
  "images": [
    {
      "fileId": "018e6b...",
      "url": "https://s3.amazonaws.com/bucket/hero.jpg",
      "thumbnailUrl": "https://s3.amazonaws.com/bucket/hero-thumb_200.jpg",
      "isPrimary": true,
      "displayOrder": 0
    }
  ]
}
```
* **Pros:** Clean, domain-focused contract. Contains the `fileId` for management, the high-res `url` for zoom, and the `thumbnailUrl` for grid views.
* **Cons:** Requires a small mapping function in the Product service layer.

## 4. Inter-Module Communication

1. **Fetching Products:** The Product Service fetches a page of products from the database, collecting all associated `file_id`s.
2. **Batch Internal Call:** The Product Service calls the `FileReadService` internally using the cross-module interface:
   ```go
   // Product Service Layer
   files, err := p.fileReadService.GetAllFiles(ctx, principal, model.GetFilesFilter{
       FileIDs:            extractedFileIDs,
       IncludeDownloadURL: true,
       IncludeVariants:    true, // Needed for thumbnails
   })
   ```
3. **Data Mapping:** The Product Service zips the retrieved `FileItem`s back onto the Product DTOs, mapping them according to the chosen response design (Option C recommended).
